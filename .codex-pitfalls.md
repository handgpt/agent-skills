# Codex Pitfalls

Concise project-specific pitfalls collected after successful fixes.
Read this before non-trivial implementation or review work.

## 2026-03-24 | Gemini advisory contracts should stay soft
- Symptom: Useful Gemini advice was dropped because headings drifted or the first line was not the exact required section heading.
- Cause: The runner and skill prompts enforced an exact markdown contract instead of only filtering empty or clearly off-task outputs.
- Rule: For Gemini advisory skills, keep the prompt structure preferred rather than exact, and validate only for empty output, obvious meta chatter, or clearly wrong advisory shape.

## 2026-03-23 | Reject stale or off-contract Gemini session outputs
- Symptom: Gemini advisory can return unrelated meta chatter or empty output when a reused session has drifted away from the current task.
- Cause: The runner reused old lane sessions too long and accepted any non-empty Gemini text without checking whether it matched the required contract.
- Rule: Bound Gemini session reuse with a TTL and validate the returned Markdown contract before accepting or reusing a session; retry fresh if a resumed session returns empty or off-contract output.

## 2026-03-21 | Gemini multi-project scope must come from workspace
- Symptom: A multi-project advisory can silently drop sibling project roots or inspect the wrong tree when it derives scope from the raw invocation directory.
- Cause: The runner used the process cwd as its boundary instead of the active Codex workspace root, so explicit project roots outside that cwd were filtered away.
- Rule: For multi-project Gemini advisory runs, derive the safety boundary from the active workspace root and only then compute the common ancestor cwd for the chosen projects.

## 2026-03-21 | Gemini prompts must declare project and scope
- Symptom: Gemini advisory occasionally mixed sibling projects or reused the wrong session when working inside a multi-project workspace.
- Cause: The prompt only declared the workspace root, and lane sessions were reused by project root alone, so sibling directories inside the same repo could share one advisory thread.
- Rule: For Gemini advisory skills, always state the project name, advisory lane, target directory, and workspace root explicitly, and scope saved sessions by the focused subdirectory instead of only by the repo root.

## 2026-03-21 | Gemini waits need visible progress and hard cap
- Symptom: Codex still treated a quiet Gemini advisory as if it had ended early, even though the runner was still waiting.
- Cause: The runner only exposed thought-based progress and used resettable timeout semantics, so long quiet queue periods looked indistinguishable from an early exit.
- Rule: For interactive Gemini skills, expose explicit wait progress to stderr and enforce a hard total timeout from prompt submission; treat thoughts as informational progress, not as unbounded timeout extensions.

## 2026-03-20 | Thought progress should keep Gemini alive
- Symptom: Interactive Gemini review exited early even after session JSON recorded new thought entries.
- Cause: The runner treated Gemini CLI process exit as final too early instead of resuming the same session while thought progress still counted as active work.
- Rule: If Gemini has produced new thought progress for the current turn, keep waiting within the inactivity timeout window and resume the same session after an early CLI exit instead of failing immediately.

## 2026-03-20 | Reset Gemini timeout on in-place thought updates
- Symptom: Interactive Gemini advisory timed out or looked stalled even though session JSON showed the same Gemini message accumulating thoughts.
- Cause: The runner only treated message-key changes as progress, so in-place updates to thoughts or tool status on an existing Gemini message did not extend the wait window.
- Rule: For interactive Gemini polling, treat semantic changes inside the latest Gemini message, especially thought and tool-call updates, as progress and reset the inactivity timeout from the latest such update.

## 2026-03-20 | Gemini interactive session completion needs strict filters
- Symptom: Interactive Gemini review can return a stale advisory or follow the wrong turn when the same sessionId is spread across multiple session-*.json files.
- Cause: Watching only the newest or currently selected chat file is insufficient because Gemini may keep prior turns in one file while appending the current turn in another file with the same sessionId.
- Rule: For interactive Gemini polling, merge all non-subagent chat files for the active sessionId, dedupe messages by id, treat only the newest user turn as authoritative, and reuse a session only when that merged final turn has a non-empty Gemini answer.

## 2026-03-20 | Gemini smoke tests need the full 20-minute timeout
- Symptom: Small Gemini smoke tests can appear to fail when a short timeout like 120s expires before the model leaves the queue.
- Cause: Gemini capacity and queue delays are currently large enough that short smoke-test timeouts produce false negatives and mislead later implementation decisions.
- Rule: For Gemini smoke tests and small advisory calls, use the same 1200-second timeout window as production runs unless a shorter limit is explicitly being tested.

## 2026-03-20 | Interactive resume can switch session files
- Symptom: A Gemini advisory can time out while the resumed session is still producing output in a newer session-*.json file.
- Cause: Interactive Gemini CLI may create a newer chat file for the same sessionId during a resumed turn, so pinning the original file misses the real output stream.
- Rule: When reusing a Gemini session interactively, always reselect the newest chat file for that sessionId instead of watching one fixed file path.

## 2026-03-20 | Prefer official Gemini JSON output
- Symptom: Gemini advisory runner became brittle because it inferred completion from interactive session files and screen state.
- Cause: The CLI already exposes an official headless JSON output path, but the runner kept scraping interactive session artifacts instead.
- Rule: For automated Gemini skills, prefer headless CLI invocation with official JSON or stream-json output over polling interactive session JSON.

## 2026-03-20 | Gemini session files are not one-to-one with session IDs
- Symptom: Runner treated the newest session JSON file as the whole resumed turn and missed the real final advisory or closed early.
- Cause: Gemini CLI chats can contain multiple session-*.json files with the same sessionId, and a resumed turn can spread its user and gemini messages across those files.
- Rule: Correlate advisory runs by sessionId plus merged message timestamps across all matching session files; do not assume one sessionId maps to one file.
