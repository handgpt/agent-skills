# Claude Code Pitfalls

Concise project-specific pitfalls collected after successful fixes.
Read this before non-trivial implementation or review work.

## 2026-04-26 | Gemini CLI v0.39+ writes .jsonl, not .json — session parsing totally breaks
- Symptom: Advisory runners return no output or hang waiting for a session file that never appears. `_latest_fresh_chat_file_since` finds nothing because it only globs `session-*.json` while Gemini now writes `session-<ts>-<shortid>.jsonl`. Even if the glob were fixed, `_load_json_file` reads the entire file as one JSON object, which fails on multi-line JSONL.
- Cause: Gemini CLI `chatRecordingService.ts` switched from a single monolithic `.json` file to an append-only JSONL format (one record per line). The first line is metadata (`sessionId`, `projectHash`, …), subsequent lines are messages, `$set` patches, or `$rewindTo` rewinds. The advisory runner was hardcoded to `.json` everywhere — globs, loader, archive.
- Rule: (1) Always glob both `session-*.json` and `session-*.jsonl` via `_all_session_file_globs()` / `_glob_session_files()`. (2) Dispatch in `_load_conversation` by `path.suffix.lower()` — `.jsonl` goes to `_load_jsonl_conversation`, everything else to legacy. (3) On a partially-written trailing line, stop parsing but salvage the successfully parsed metadata and messages rather than returning `None`. (4) When upstream changes the file format, check BOTH runner copies (`claude-code/` and `codex/`) plus the test file.

## 2026-04-10 | Gemini advisory contamination via shared project chats dir (even without --resume)
- Symptom: gemini-review returned pointer-model / BleHidController content from a previous v0.50.8 review session instead of analyzing the current update-migration brief. Despite `--resume` already being disabled (2026-04-07 fix), the runner still returned an assistant reply from an unrelated historical session.
- Cause: `_run_interactive_attempt` initialized `baseline_message_keys = set()` whenever `resumed_session_id == ""` (which is always now). The runner's `_latest_fresh_chat_file_since` would resolve a session file in the shared per-project chats dir `~/.gemini/tmp/<short>/chats/`, `_merged_session_messages` would return every message in that file, and because baseline was empty, all pre-existing historical messages passed into `new_messages`. `_interactive_outcome` then returned an old assistant reply as this invocation's result without Gemini ever processing the new brief. An index-based "anchor baseline at run_marker position" fix would NOT work because `_message_timestamp_sort_value` returns `float('inf')` for messages with missing timestamps, pushing historical messages *after* our anchor in the merged list and evading any slice-based exclusion.
- Rule: For advisory lanes with a shared per-project chats dir, (1) archive every pre-existing `session-*.json` into a timestamped `.advisory-archive-<UTC>/` subdirectory before spawning Gemini so the dir is empty at spawn time (gated by `CLAUDE_GEMINI_ADVISORY_ARCHIVE_STALE_CHATS`, default on), and (2) snapshot every `_message_identity` present in the chats dir at spawn time and seed `baseline_message_keys` from that snapshot. The snapshot is order-independent and does not depend on sort-order heuristics or run-marker position, so it is robust against `float('inf')` timestamp edge cases, mtime touches by external processes, and Gemini-CLI file reuse.

## 2026-04-10 | POSIX double-fork daemon must pipe PID back and preserve cwd
- Symptom: Initial `--daemon` mode in gemini_runner.py printed a dead PID (the first child's, which exits immediately during the double-fork) and broke all relative path resolution by calling `os.chdir("/")` before argparse paths were resolved.
- Cause: Textbook daemon recipes include `chdir("/")` because classic long-lived daemons should not hold a mount, and they print the first fork's PID assuming that's the daemon. Both are wrong for a short-lived tool daemon that (a) must operate on the original workspace cwd and (b) needs a tracking PID for Monitor tool / kill / wrappers.
- Rule: For tool-style daemons that operate on the caller's workspace, (1) create an `os.pipe()` before the first fork and have the grandchild write `os.getpid()` back to the original parent so the printed PID is the real one, (2) do NOT `os.chdir("/")` — preserve cwd so relative `--brief-file` / `--context-file` / `--project-root` keep resolving, (3) guard `os.close(fd)` with `if fd > 2` because `os.open` can return 0/1/2 if stdio was already closed, and (4) reap the first child with `waitpid` to avoid a zombie.

## 2026-04-10 | install.sh .claude/ output must be gitignored
- Symptom: `.claude/commands/*.md` created by install.sh contains user-local absolute paths (`/Users/<name>/...`) after `$AGENT_SKILLS_DIR` substitution, but was not in `.gitignore`, risking accidental commit of per-machine artifacts.
- Cause: install.sh produces a substituted copy of the slash commands into `.claude/commands/` but the gitignore was never updated to exclude the output directory.
- Rule: Whenever an install script materializes per-machine files into the source tree, its output directory must be added to `.gitignore` in the same change.

## 2026-04-07 | Gemini session reuse causes empty responses and stale context
- Symptom: 3 out of 4 resumed Gemini sessions returned empty content (content='') despite receiving detailed unique briefs. Different advisory tasks (diagnostic, design review, post-fix review) all routed to the same sessionId produced no output.
- Cause: Session reuse via --resume appended new briefs into an existing session with 22+ messages of unrelated context. Gemini either returned empty responses or analyzed old session content instead of the new brief.
- Rule: Never reuse Gemini sessions across advisory invocations. Each advisory task must start a fresh session. Within-run resume (for mid-turn crashes) is acceptable but cross-invocation resume is not.

## 2026-03-31 | install.sh AGENT_SKILLS_DIR resolved two levels up instead of one
- Symptom: Skills scripts fail with 'No such file or directory' because installed command paths are missing the agent-skills/ directory component
- Cause: install.sh used CLAUDE_CODE_DIR/../.. (two parent levels) to resolve AGENT_SKILLS_DIR instead of CLAUDE_CODE_DIR/.. (one level), so $AGENT_SKILLS_DIR pointed to the workspace root instead of the agent-skills repo root
- Rule: When computing parent paths in install scripts, count the directory levels carefully. Verify resolved paths against actual filesystem structure before substitution.

## 2026-03-30 | Never substitute user-specified parameter values
- Symptom: User specified gemini-3.1-pro-preview as the model but code was written with gemini-2.5-pro instead
- Cause: Agent substituted a model name from training knowledge instead of using the exact value the user provided, treating the user input as an error to correct
- Rule: When the user specifies an exact parameter value (model name, version, URL, etc), use it verbatim. Never silently replace it with what you think is correct. If you believe the value may be wrong, ask first

## 2026-03-30 | Gemini cross-directory code attribution hallucination
- Symptom: Gemini structural review reported numerous Codex references, fabricated variable names (CLAUDE_GEMINI_CLI_USER_OAUTH_TOKEN), and non-existent file names (codex-gemini-run-state.json) in claude-code/ files that actually had zero Codex references
- Cause: Gemini CLI runs with full filesystem access (--approval-mode yolo) from the workspace root. When reviewing claude-code/, it browsed the sibling codex/ directory and misattributed codex/ code patterns to claude-code/ files
- Rule: When running Gemini advisory on a subdirectory, set --project-root to that subdirectory specifically and add explicit instructions in the brief to NOT inspect sibling directories. Also verify Gemini findings against grep before acting on them
